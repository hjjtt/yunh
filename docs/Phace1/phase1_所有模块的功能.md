我结合你项目里每个模块的**实际文件、代码、依赖关系**，给你逐个讲解每个模块的作用，你可以对照着自己的项目结构理解：

***

## 📦 项目模块逐一说解

### 总体结构回顾

```
yunh-cloud (父工程)
├─ yunh-basic/          🔹 基础通用层
├─ yunh-pojo/           🔹 数据模型层
├─ yunh-api/            🔹 接口契约层
├─ yunh-service/        🔹 业务服务层
└─ yunh-support/        🔹 架构支撑层
```

***

### 1. 父工程：`yunh-cloud` (根目录pom.xml)

#### ✅ 作用：项目的"总指挥部"，统一管理所有子模块

#### 📝 包含内容：

- `<modules>`：声明所有子模块列表，Maven编译时会按这个顺序编译
- `<properties>`：定义所有版本号（Spring Boot、Spring Cloud、各种组件版本）
- `<dependencyManagement>`：所有依赖的版本声明，子模块引用时不需要写版本
- `<dependencies>`：所有子模块都要用的公共依赖（Lombok、Fastjson）
- `<build>`：统一编译配置（JDK版本、编码、插件）

#### 🔗 依赖关系：无上层依赖，是所有子模块的父项目

#### 💡 实际价值：整个项目的"配置中心"，升级版本只需要改这里，所有子模块自动生效

***

### 2. 基础通用层：`yunh-basic/`

#### ✅ 作用：整个项目的"工具库"，所有模块都可以依赖它，放和业务无关的通用能力

#### 📝 包含2个子模块：

| 子模块                     | 作用                                                | 包含内容                                                                                         |
| :---------------------- | :------------------------------------------------ | :------------------------------------------------------------------------------------------- |
| `yunh-basic-dependency` | 空的依赖管理模块，用来统一管理依赖（有的项目会把所有第三方依赖都放这里，父工程只管理内部模块版本） | 只有pom.xml，没有代码                                                                               |
| `yunh-basic-common`     | 通用工具类、全局处理                                        | - 统一响应类：`Result`、`ResultCode`- 全局异常处理器：`GlobalExceptionHandler`- 自定义业务异常：`BusinessException` |

#### 🔗 依赖关系：没有内部依赖，只依赖第三方包（Spring、Lombok等）

#### 💡 实际价值：

- 你加一个`IdWorker`分布式ID生成器、加一个`JwtUtils`工具类，都放这里，所有服务都能用
- 不用每个服务都写一套异常处理、响应类

***

### 3. 数据模型层：`yunh-pojo/`

#### ✅ 作用："数据契约"，定义所有数据库表对应的实体类、DTO、VO，是微服务之间传递数据的标准格式

#### 📝 包含3个子模块：

| 子模块                | 作用       | 包含内容                                      |
| :----------------- | :------- | :---------------------------------------- |
| `yunh-pojo-common` | 公共数据模型   | 分页请求DTO、通用返回DTO等所有服务都会用到的数据结构             |
| `yunh-pojo-user`   | 用户服务数据模型 | `User.java` → 对应数据库`t_user`表，包含用户所有字段     |
| `yunh-pojo-course` | 课程服务数据模型 | `Course.java` → 对应数据库`t_course`表，包含课程所有字段 |

#### 🔗 依赖关系：只依赖`yunh-basic-common`和MyBatis Plus（因为实体类要用到`@TableName`、`@TableId`注解）

#### 💡 实际价值：

- 纯Java类，没有任何业务逻辑，所以可以放心给任何模块引用
- 服务之间调用接口返回`User`对象，调用方和提供方都用同一个类，不需要各自定义一遍，避免数据结构不一致

***

### 4. 接口契约层：`yunh-api/`

#### ✅ 作用：服务对外暴露的"接口说明书"，其他服务要调用这个服务，只需要引入这个模块就可以

#### 📝 包含2个子模块：

| 子模块               | 作用         | 包含内容                                                            |
| :---------------- | :--------- | :-------------------------------------------------------------- |
| `yunh-api-user`   | 用户服务对外接口定义 | 只写接口，不写实现：`public interface UserApi { User getById(Long id); }` |
| `yunh-api-course` | 课程服务对外接口定义 | 只写接口，不写实现：`public interface CourseApi { List<Course> list(); }` |

#### 🔗 依赖关系：依赖对应的pojo模块（比如`yunh-api-user`依赖`yunh-pojo-user`，因为接口要返回User对象）

#### 💡 实际价值：

- 课程服务要调用用户服务，只需要引入`yunh-api-user`这个jar包（几KB），不需要把整个用户服务的代码都引进来
- 基于这个接口可以用Feign做远程调用，非常方便，接口定义修改了，调用方自动感知

***

### 5. 业务服务层：`yunh-service/`

#### ✅ 作用：具体业务逻辑实现，就是实际运行的微服务

#### 📝 包含3个子模块：

| 子模块                   | 作用             | 包含内容                                                                                    |
| :-------------------- | :------------- | :-------------------------------------------------------------------------------------- |
| `yunh-service-common` | 服务通用配置         | 所有服务都会用到的配置：MyBatis Plus配置、Nacos配置、拦截器等                                                 |
| `yunh-service-user`   | 用户服务实现（端口9001） | - Controller、Service、Mapper层实现- `application.yml`配置- 启动类`UserServiceApplication.java`   |
| `yunh-service-course` | 课程服务实现（端口9002） | - Controller、Service、Mapper层实现- `application.yml`配置- 启动类`CourseServiceApplication.java` |

#### 🔗 依赖关系：依赖对应的api模块、`yunh-service-common`、`yunh-basic-common`

#### 💡 实际价值：

- 这里才是真正写业务代码的地方，用户注册、登录、课程查询这些业务逻辑都在这里实现
- 启动后会注册到Nacos，供其他服务调用，也可以被前端直接调用

***

### 6. 架构支撑层：`yunh-support/`

#### ✅ 作用：微服务架构的基础设施组件，不属于业务，但是整个架构离不开

#### 📝 包含1个子模块：

| 子模块                    | 作用    | 包含内容                             |
| :--------------------- | :---- | :------------------------------- |
| `yunh-service-gateway` | API网关 | 所有请求的统一入口，负责路由转发、权限校验、限流、日志统一打印等 |

#### 🔗 依赖关系：依赖`yunh-basic-common`和Spring Cloud Gateway

#### 💡 实际价值：

- 前端不需要知道每个服务的端口，只需要调用网关地址，网关自动转发到对应的服务
- 登录校验、权限判断这些公共逻辑可以统一在网关做，不用每个服务都写一遍

***

## 🎯 模块调用关系示例

举个例子：**前端查询课程列表**

1. 前端请求 → `yunh-service-gateway`（网关）
2. 网关转发到 → `yunh-service-course`（课程服务）
3. 课程服务如果需要查询创建课程的用户信息 → 调用`yunh-api-user`的接口
4. 底层通过Feign远程调用 → `yunh-service-user`（用户服务）
5. 用户服务返回`User`对象 → 定义在`yunh-pojo-user`中
6. 所有接口统一返回`Result`对象 → 定义在`yunh-basic-common`中

这样整个调用链路的所有数据结构、响应格式都是统一的，非常清晰。
