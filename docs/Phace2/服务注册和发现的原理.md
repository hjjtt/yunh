## 🎯 什么是服务注册与发现？

### 先看生活化类比：外卖平台

| 角色       | 对应技术组件              | 作用                     |
| :------- | :------------------ | :--------------------- |
| 外卖平台     | Nacos 注册中心          | 统一管理所有入驻商家的信息          |
| 商家A（卖奶茶） | yunh-service-user   | 提供用户查询服务，启动时去Nacos"入驻" |
| 商家B（卖快餐） | yunh-service-course | 提供课程查询服务，启动时去Nacos"入驻" |
| 点外卖的用户   | 前端/网关/其他服务          | 要调用服务时，先去Nacos找对应服务的地址 |

你不用记商家的具体地址，只需要知道"我要找卖奶茶的商家"，平台会给你返回所有营业的奶茶店地址，你直接去买就行。这就是服务注册与发现的核心思想。

***

## 🔄 核心原理（4步流程）

### 1. 「服务注册」：服务启动时主动上报信息

你的`yunh-service-user`启动的时候，会自动给Nacos发请求，上报自己的信息：

```
服务名：yunh-service-user
IP：192.168.1.100
端口：9001
健康状态：正常
```

Nacos把这些信息存在自己的服务列表里，相当于商家在平台"入驻成功"。

✅ **你项目里对应的配置**：
在每个服务的`application.yml`里都有：

```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848 # 告诉服务Nacos的地址
```

启动类上的`@SpringBootApplication`已经默认开启了服务注册功能，不需要额外加注解。

***

### 2. 「心跳续约」：定期报平安

服务启动后，**每隔30秒**会给Nacos发一次"心跳"请求，告诉Nacos"我还在运行，别把我踢了"。
如果Nacos**连续90秒**都没收到某个服务的心跳，就认为这个服务已经挂了，自动把它从服务列表里删除。

相当于商家每隔半小时给平台发个消息"我还在营业"，如果半天没消息，平台就把这个商家下架。

***

### 3. 「服务发现」：消费者找服务地址

假设你的`yunh-service-course`要调用`yunh-service-user`的接口：

1. 课程服务启动时，会先从Nacos拉取所有服务的列表，缓存到本地
2. 要调用用户服务时，不需要问Nacos，直接从本地缓存里找`yunh-service-user`的所有实例地址
3. 用负载均衡策略（比如轮询）选一个地址，直接发起HTTP调用

✅ **为什么要缓存到本地？** 不用每次调用都请求Nacos，性能更高，就算Nacos挂了，短时间内服务之间还能正常调用。

***

### 4. 「服务剔除」：下线自动感知

- 服务正常下线时，会主动给Nacos发请求"我要下线了"，Nacos直接把它从列表里移除
- 服务异常挂掉时，Nacos等不到心跳，90秒后自动剔除
- 剔除后，Nacos会通知所有消费者更新本地缓存，下次调用就不会走到挂掉的服务上了

## 🎯 为什么要用服务注册与发现？

### 传统单体架构不用这个的原因：

所有服务都在一个项目里，直接本地方法调用就行，不需要知道对方的IP和端口。

### 微服务架构必须用的原因：

1. **动态IP的问题**：微服务部署在云服务器上，IP是动态分配的，你不可能每次服务重启都改配置
2. **多实例负载均衡**：一个服务可以部署多个实例（比如用户服务部署3台），注册中心可以自动返回所有实例地址，消费者可以做负载均衡
3. **自动故障剔除**：某个实例挂了，注册中心自动把它踢掉，不会把请求转发到挂掉的服务上
4. **服务伸缩方便**：要加机器只需要启动新的服务实例，自动注册到Nacos，消费者自动就能感知到，不需要改任何配置

简单说：没有服务注册与发现，微服务架构根本跑不起来，Nacos就是微服务架构的"通讯录"。
