## 📚 核心概念

### 1. 为什么需要分布式事务？

#### 场景：用户选课

```
┌─────────────────────────────────────────────────────────────┐
│                    用户发起选课请求                           │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  ┌─────────────────┐         ┌─────────────────┐            │
│  │  User 服务       │         │  Course 服务     │            │
│  │  (9001)        │         │  (9002)        │            │
│  │                 │         │                 │            │
│  │ 1. 添加选课记录  │         │ 2. 扣减课程库存  │            │
│  │    ✅ 成功      │         │    ❌ 失败！     │            │
│  └─────────────────┘         └─────────────────┘            │
│                                                              │
│  ❌ 问题：User 服务已提交，Course 服务失败                     │
│     数据不一致：用户有选课记录，但课程库存没扣减               │
└─────────────────────────────────────────────────────────────┘
```

#### 传统解决方案的问题

| 方案 | 问题 |
|------|------|
| 本地事务 | 只能保证单个数据库一致，跨服务无效 |
| 手动补偿 | 代码复杂，容易遗漏，难以维护 |
| 消息队列 | 最终一致性，实现复杂，有延迟 |

### 2. CAP 理论

```
┌─────────────────────────────────────────────────────────────┐
│                        CAP 三角                              │
│                                                              │
│                      Consistency (一致性)                    │
│                          /    \                              │
│                         /      \                             │
│                        /        \                            │
│                       /          \                           │
│                      /            \                          │
│                     /              \                         │
│                    /                \                        │
│                   /                  \                       │
│                  /                    \                      │
│                 /                      \                     │
│                /                        \                    │
│               /                          \                   │
│              /                            \                  │
│             /                              \                 │
│            /                                \                │
│           /                                  \               │
│          /                                    \              │
│         /                                      \             │
│        /                                        \            │
│       /                                          \           │
│      /                                            \          │
│     /                                              \         │
│    /                                                \        │
│   /                                                  \       │
│  /                                                    \      │
│ /                                                      \     │
│/                                                        \    │
└─────────────────────────────────────────────────────────────┘
Availability (可用性)              Partition Tolerance (分区容错性)
```

**CAP 三者不可兼得，最多满足两个：**

| 组合 | 说明 | 典型场景 |
|------|------|---------|
| **CP** | 一致性 + 分区容错 | 银行转账、支付（牺牲可用性） |
| **AP** | 可用性 + 分区容错 | 社交动态、商品浏览（牺牲一致性） |
| ~~CA~~ | 一致性 + 可用性 | 不可能（分布式系统必须有 P） |

**微服务的选择：** 通常是 **AP + 最终一致性**

---

### 3. BASE 理论

BASE 是 CAP 中 AP 方案的延伸：

| 要素 | 英文 | 说明 |
|------|------|------|
| **基本可用** | Basically Available | 系统大部分时间可用，允许部分功能不可用 |
| **软状态** | Soft State | 允许中间状态，不要求实时一致 |
| **最终一致** | Eventually Consistent | 经过一段时间后，数据最终会一致 |

**例子：** 淘宝购物
- 下单后库存"暂时"扣减（软状态）
- 支付成功后库存"最终"扣减（最终一致）
- 期间可能看到"库存不足"（基本可用）

---

### 4. Seata 是什么？

**Seata** = **S**imple **E**xtensible **A**utonomous **T**ransaction **A**rchitecture

阿里巴巴开源的**分布式事务解决方案**，支持多种事务模式：

| 模式 | 说明 | 性能 | 一致性 | 适用场景 |
|------|------|------|--------|---------|
| **AT** | 自动补偿（两阶段提交） | ⭐⭐⭐ | 强一致 | 大多数场景（推荐） |
| **TCC** | 手动补偿（Try-Confirm-Cancel） | ⭐⭐⭐⭐ | 强一致 | 高性能要求 |
| **Saga** | 长事务（状态机） | ⭐⭐⭐⭐ | 最终一致 | 长流程业务 |
| **XA** | 数据库原生 XA 协议 | ⭐⭐ | 强一致 | 传统系统 |

**本项目选择：AT 模式**（无侵入、易使用）

---

### 5. Seata AT 模式原理

```
┌─────────────────────────────────────────────────────────────┐
│                    第一阶段：执行 + 记录                      │
│                                                              │
│  1. Seata 拦截业务 SQL                                        │
│  2. 查询更新前的数据（before image）                          │
│  3. 查询更新后的数据（after image）                           │
│  4. 提交本地事务                                             │
│  5. 将 before/after image 写入 undo_log 表                    │
│  6. 向 TC 报告执行结果                                        │
│                                                              │
│  ✅ 优点：业务代码无感知，自动处理                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │   TC 协调器      │
                    │  (Seata Server) │
                    └─────────────────┘
                              │
                    ┌─────────┴─────────┐
                    │                   │
                    ▼                   ▼
          ┌─────────────────┐ ┌─────────────────┐
          │   全部成功       │ │   部分失败       │
          │                 │ │                 │
          │  第二阶段：提交   │ │  第二阶段：回滚  │
          │  - 异步删除      │ │  - 根据 undo_   │
          │    undo_log     │ │    log 反向补偿  │
          │  - 无业务感知    │ │  - 恢复 before  │
          │                 │ │    image 数据    │
          └─────────────────┘ └─────────────────┘
```

```sql
-- 在 MySQL 中执行
CREATE DATABASE IF NOT EXISTS seata DEFAULT CHARACTER SET utf8mb4;

USE seata;

-- 全局事务表
CREATE TABLE IF NOT EXISTS `global_table`
(
    `xid`                       VARCHAR(128) NOT NULL,
    `transaction_id`            BIGINT,
    `status`                    TINYINT      NOT NULL,
    `application_id`            VARCHAR(32),
    `transaction_service_group` VARCHAR(32),
    `transaction_name`          VARCHAR(128),
    `timeout`                   INT,
    `begin_time`                BIGINT,
    `application_data`          VARCHAR(2000),
    `gmt_create`                DATETIME,
    `gmt_modified`              DATETIME,
    PRIMARY KEY (`xid`),
    KEY `idx_gmt_modified_status` (`gmt_modified`, `status`),
    KEY `idx_transaction_id` (`transaction_id`)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4;

-- 分支事务表
CREATE TABLE IF NOT EXISTS `branch_table`
(
    `branch_id`         BIGINT       NOT NULL,
    `xid`               VARCHAR(128) NOT NULL,
    `transaction_id`    BIGINT,
    `resource_group_id` VARCHAR(32),
    `resource_id`       VARCHAR(256),
    `branch_type`       VARCHAR(8),
    `status`            TINYINT,
    `client_id`         VARCHAR(64),
    `application_data`  VARCHAR(2000),
    `gmt_create`        DATETIME,
    `gmt_modified`      DATETIME,
    PRIMARY KEY (`branch_id`),
    KEY `idx_xid` (`xid`)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4;

-- 锁表
CREATE TABLE IF NOT EXISTS `lock_table`
(
    `row_key`        VARCHAR(128) NOT NULL,
    `xid`            VARCHAR(96),
    `transaction_id` BIGINT,
    `branch_id`      BIGINT       NOT NULL,
    `resource_id`    VARCHAR(256),
    `table_name`     VARCHAR(32),
    `pk`             VARCHAR(36),
    `gmt_create`     DATETIME,
    `gmt_modified`   DATETIME,
    PRIMARY KEY (`row_key`),
    KEY `idx_branch_id` (`branch_id`)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4;
```

#### 6.4.2 User 服务：添加全局事务注解

```java
// yunh-service-user/src/main/java/.../service/impl/UserCourseServiceImpl.java
package com.yunh.user.service.impl;

import com.yunh.api.client.CourseClient;
import com.yunh.pojo.dto.SelectCourseDTO;
import com.yunh.pojo.entity.UserCourse;
import com.yunh.user.mapper.UserCourseMapper;
import io.seata.spring.annotation.GlobalTransactional;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Slf4j
@Service
@RequiredArgsConstructor
public class UserCourseServiceImpl implements UserCourseService {
    
    private final UserCourseMapper userCourseMapper;
    private final CourseClient courseClient;
    
    /**
     * 用户选课（分布式事务）
     * 
     * @GlobalTransactional 标注全局事务
     * - name: 事务名称（便于监控）
     * - rollbackFor: 哪些异常触发回滚
     */
    @Override
    @GlobalTransactional(name = "yunh-select-course", rollbackFor = Exception.class)
    @Transactional(rollbackFor = Exception.class)
    public void selectCourse(SelectCourseDTO dto) {
        Long userId = dto.getUserId();
        Long courseId = dto.getCourseId();
        
        log.info("开始选课事务：userId={}, courseId={}", userId, courseId);
        
        // 1. 添加选课记录（本地事务）
        UserCourse userCourse = new UserCourse();
        userCourse.setUserId(userId);
        userCourse.setCourseId(courseId);
        userCourse.setStatus(0); // 待支付
        userCourseMapper.insert(userCourse);
        
        log.info("选课记录已添加：{}", userCourse.getId());
        
        // 2. 扣减课程库存（远程调用）
        // 如果此步骤失败，会触发全局回滚，步骤 1 的数据也会回滚
        courseClient.decreaseStock(courseId, 1);
        
        log.info("课程库存已扣减：courseId={}", courseId);
        
        // 3. 模拟异常（测试用）
        // if (true) {
        //     throw new RuntimeException("选课失败，触发回滚");
        // }
        
        log.info("选课事务完成：userId={}, courseId={}", userId, courseId);
    }
}
```

```java
// yunh-service-course/src/main/java/.../service/impl/CourseStockServiceImpl.java
package com.yunh.course.service.impl;

import com.yunh.course.mapper.CourseMapper;
import com.yunh.course.service.CourseStockService;
import com.yunh.pojo.entity.Course;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Slf4j
@Service
@RequiredArgsConstructor
public class CourseStockServiceImpl implements CourseStockService {
    
    private final CourseMapper courseMapper;
    
    /**
     * 扣减课程库存
     * 注意：不需要 @GlobalTransactional，由调用方发起全局事务
     */
    @Override
    @Transactional(rollbackFor = Exception.class)
    public void decreaseStock(Long courseId, Integer count) {
        log.info("开始扣减库存：courseId={}, count={}", courseId, count);
        
        // 1. 查询课程
        Course course = courseMapper.selectById(courseId);
        if (course == null) {
            throw new RuntimeException("课程不存在");
        }
        
        // 2. 检查库存
        if (course.getStock() < count) {
            throw new RuntimeException("库存不足");
        }
        
        // 3. 扣减库存
        int updated = courseMapper.decreaseStock(courseId, count);
        if (updated == 0) {
            throw new RuntimeException("扣减库存失败");
        }
        
        log.info("库存扣减成功：courseId={}, 剩余库存={}", courseId, course.getStock() - count);
    }
}
```

### 步骤 6.5：配置 Gateway 传递 Seata 上下文（预计 15 分钟）

#### 6.5.1 在 Gateway 中添加 Seata 过滤器

```java
// yunh-support/yunh-service-gateway/src/main/java/.../config/SeataGatewayConfig.java
package com.yunh.gateway.config;

import io.seata.core.context.RootContext;
import org.springframework.cloud.gateway.filter.GatewayFilterChain;
import org.springframework.cloud.gateway.filter.GlobalFilter;
import org.springframework.core.Ordered;
import org.springframework.http.server.reactive.ServerHttpRequest;
import org.springframework.stereotype.Component;
import org.springframework.web.server.ServerWebExchange;
import reactor.core.publisher.Mono;

/**
 * Seata 事务 ID 传递过滤器
 * 将 Gateway 接收到的 XID 传递给下游服务
 */
@Component
public class SeataGatewayFilter implements GlobalFilter, Ordered {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        
        // 从请求头获取 XID（如果 Gateway 是事务发起方）
        String xid = request.getHeaders().getFirst(RootContext.KEY_XID);
        
        if (xid != null && !xid.isEmpty()) {
            // 将 XID 传递给下游服务
            ServerHttpRequest mutatedRequest = request.mutate()
                .header(RootContext.KEY_XID, xid)
                .build();
            return chain.filter(exchange.mutate().request(mutatedRequest).build());
        }
        
        return chain.filter(exchange);
    }
    
    @Override
    public int getOrder() {
        return -200; // 高优先级，在业务过滤器之前执行
    }
}
```

## ⚠️ 注意事项

### 1. 版本兼容性

| 组件 | 版本 |
|------|------|
| Spring Boot | 2.2.5.RELEASE |
| Spring Cloud | Hoxton.SR3 |
| Spring Cloud Alibaba | 2.2.1.RELEASE |
| **Seata** | **1.4.2** |

### 2. 事务超时

默认全局事务超时时间为 60 秒，超时会自动回滚：

```yaml
seata:
  tm:
    default-global-transaction-timeout: 60000  # 60 秒
```

### 3. 隔离级别

AT 模式默认**全局读已提交**隔离级别：
- 全局未提交的事务，其他事务读不到
- 通过 `lock_table` 实现全局锁

### 4. 性能影响

| 操作 | 性能影响 |
|------|---------|
| before/after image | 额外查询 |
| undo_log 写入 | 额外写入 |
| 全局锁 | 并发降低 |

**优化建议：**
- 事务范围尽量小
- 避免在事务中做 HTTP 请求
- 大事务考虑 Saga 模式

### 5. 常见问题

| 问题 | 解决方法 |
|------|---------|
| 事务不回滚 | 检查异常是否被捕获、rollbackFor 配置 |
| XID 不传递 | 检查 Feign 拦截器、Header 配置 |
| 死锁 | 检查 SQL 执行顺序、添加超时 |
| undo_log 不删除 | 检查事务是否正常提交/回滚 |
