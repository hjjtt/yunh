## 2. 当前总体判断

项目当前已经具备完整的学习型微服务骨架：

- Spring Cloud 多模块后端
- 微信小程序前端
- Vue 管理端
- Nacos 注册中心与配置中心
- Gateway 网关
- JWT 鉴权
- OpenFeign 服务调用
- Sentinel 限流
- Seata 分布式事务
- RocketMQ 消息队列

当前最主要的风险已经不再是“能不能运行”，而是以下四类：

- **敏感信息暴露**
- **接口越权与鉴权边界不清**
- **环境与文档口径不一致**
- **测试与回归验证体系不足**

补充说明：

- 指定后端模块已执行编译检查，结果为 **BUILD SUCCESS**
- 当前问题更偏向 **运行期安全风险** 和 **工程治理问题**

## 3. 项目结构概览

### 3.1 后端结构

根聚合工程见 [pom.xml](D:\vis\yunh\pom.xml:1)，主要模块包括：

- `yunh-basic`
- `yunh-pojo`
- `yunh-api`
- `yunh-service`
- `yunh-support`
- `yunh-auth`

后端技术基线：

- Spring Boot 2.2.5.RELEASE
- Spring Cloud Hoxton.SR3
- JDK 1.8
- MyBatis Plus
- JWT
- Seata
- RocketMQ
- Sentinel

### 3.2 前端结构

前端包含两个独立入口：

- 微信小程序：`yunh-mini`
- Vue 管理端：`yunh-admin`

小程序主入口见 [app.json](D:\vis\yunh\yunh-mini\miniprogram\app.json:1)，已包含：

- 学员端页面
- 教师端页面
- 课程、订单、个人中心等分包页面

### 3.3 核心业务链路

当前项目主要链路如下：

- 登录链路：`yunh-mini / yunh-admin -> Gateway -> yunh-auth`
- 课程链路：`Gateway -> yunh-service-course`
- 订单链路：`Gateway -> yunh-service-order`
- 支付链路：`Gateway -> yunh-service-pay`
- 互动链路：`Gateway -> yunh-service-interaction`

## 4. 已确认问题

## 4.1 严重问题

### 4.1.1 用户敏感信息暴露

**问题等级：严重**

**问题表现：**

- 用户实体包含 `password`
- 用户公开接口直接返回 `User` 实体
- 小程序登录后会把完整用户对象写入本地缓存
- 匿名路径还暴露了部分用户查询能力

**涉及文件：**

- [User.java](D:\vis\yunh\yunh-pojo\yunh-pojo-user\src\main\java\com\yunh\user\pojo\User.java:31)
- [UserController.java](D:\vis\yunh\yunh-service\yunh-service-user\src\main\java\com\yunh\user\controller\UserController.java:45)
- [AuthGlobalFilter.java](D:\vis\yunh\yunh-support\yunh-service-gateway\src\main\java\com\yunh\gateway\filter\AuthGlobalFilter.java:23)
- [login.js](D:\vis\yunh\yunh-mini\miniprogram\pages\auth\login.js:85)
- [session.js](D:\vis\yunh\yunh-mini\miniprogram\store\session.js:1)

**风险说明：**

- 可能暴露密码哈希
- 可能暴露邮箱、手机号等个人信息
- 会在客户端本地留下不必要的敏感数据副本

## 5. 全局结构与工程问题

### 5.1 文档与实际实现不一致

**问题等级：高**

**表现：**

- 根 [README.md](D:\vis\yunh\README.md:1) 中的端口、阶段说明、文件引用已明显过时
- 小程序 README 与实际 `AppID` 配置不一致
- `docs` 目录中文档命名不统一，部分引用路径已失效

---

### 5.2 环境配置口径冲突

**问题等级：高**

**表现：**

- [database/init.sql](D:\vis\yunh\database\init.sql:1) 使用多库模型
- Nacos 本地配置快照也是多库模型
- `publish-nacos-configs.ps1` 又存在发布成单库模型的风险

**风险说明：**

- 同一项目存在多套“真实配置”
- 很容易导致本地环境与文档、脚本、线上认知互相打架

---

### 5.3 业务链路存在双轨实现

**问题等级：中**

**表现：**

- 一条是小程序直连 `order + pay`
- 另一条是用户服务里的 `selectCourse + Seata + MQ`

**风险说明：**

- 后续维护时容易出现逻辑分叉
- 不利于确认“正式业务链路”到底是哪一条

---

### 5.4 仓库洁净度较差

**问题等级：中**

**表现：**

- `target/`
- `node_modules/`
- `log/`
- `.idea/`
- `.vscode/`
- 历史中间件目录
- stray `.class`

等内容混在仓库工作目录中。

**风险说明：**

- 干扰巡检、构建和协作
- 增大误判概率

---

### 5.5 自动化测试基本缺失

**问题等级：高**

**表现：**

- 当前项目缺少稳定的单元测试、集成测试和链路回归测试
- 很多验证依赖手工联调和日志回看

**风险说明：**

- 修复后不容易确认是否引入回归
- 分布式事务、鉴权、消息链路难以稳定验证

---

## 8. 结论

当前项目在“功能骨架”层面已经比较完整，但在“安全边界”和“工程治理”层面仍有明显缺口。

现阶段的核心任务不再是继续堆功能，而是把以下基础做扎实：

- **敏感数据不外泄**
- **接口权限边界可控**
- **环境配置唯一可信**
- **修复后可验证、可回归**

如果这一轮治理完成，项目的稳定性和后续迭代效率都会明显提升。
