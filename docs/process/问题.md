# 项目问题清单

## 文档说明

本文档基于当前阶段已完成的项目巡检结果整理，主要来源于：

- 项目整体结构梳理
- 主线程工程检查
- 健壮性检查代理返回结果

当前文档记录的是**已经确认的问题**，其中“代码错误检查代理”和“项目全局检查代理”尚未完成最终回传，因此后续仍可能继续补充。

## 当前总体判断

项目整体功能覆盖较完整，已经具备以下核心能力：

- Spring Cloud 多模块微服务架构
- Nacos 注册中心与配置中心
- Gateway 网关
- JWT 鉴权
- OpenFeign 服务调用
- Sentinel 限流
- Seata 分布式事务
- RocketMQ 消息队列
- Vue 3 管理端
- 微信小程序端

但当前主要风险不在“是否能跑起来”，而在以下几个方面：

- **跨服务一致性不足**
- **失败路径处理不完整**
- **接口入参边界校验薄弱**
- **异常处理存在假成功风险**
- **自动化测试基本缺失**

## 一、高优先级问题

### 1. 选课主流程未校验远程扣库存结果

**问题等级：严重**

**涉及文件：**

- [UserCourseService.java](D:\vis\yunh\yunh-service\yunh-service-user\src\main\java\com\yunh\user\service\UserCourseService.java:96)
- [CourseFeignClientFallback.java](D:\vis\yunh\yunh-api\yunh-api-course\src\main\java\com\yunh\api\client\fallback\CourseFeignClientFallback.java:41)

**问题描述：**

在用户选课主流程中，远程调用课程服务扣减库存时，如果课程服务不可用、Feign 触发 fallback、或者 Sentinel 发生降级，`decreaseStock()` 可能返回失败结果，但当前流程没有严格校验返回值是否成功。

这会导致本地选课记录继续写入，后续通知消息和延迟取消消息照常发送。

**潜在后果：**

- 选课记录已创建
- 课程库存未扣减
- 本地事务与远程服务状态不一致
- 后续补偿和对账难度增加

**建议修复方向：**

- 对远程调用结果做 `null` 和成功状态双重校验
- 若远程扣库存失败，必须立即抛出异常
- 触发本地事务和 Seata 全局事务回滚

### 2. 支付回调存在跨服务状态错位风险

**问题等级：严重**

**涉及文件：**

- [PaymentServiceImpl.java](D:\vis\yunh\yunh-service\yunh-service-pay\src\main\java\com\yunh\pay\service\impl\PaymentServiceImpl.java:73)
- [PaymentServiceImpl.java](D:\vis\yunh\yunh-service\yunh-service-pay\src\main\java\com\yunh\pay\service\impl\PaymentServiceImpl.java:76)
- [PaymentServiceImpl.java](D:\vis\yunh\yunh-service\yunh-service-pay\src\main\java\com\yunh\pay\service\impl\PaymentServiceImpl.java:103)

**问题描述：**

支付回调处理中，当前逻辑是先同步远程订单状态，再更新本地支付单状态。如果远程订单更新成功，但本地支付单更新失败，就会出现状态不一致。

**潜在后果：**

- 订单服务显示“已支付”
- 支付服务本地记录仍显示“未支付”
- 影响幂等、退款、对账、补偿处理

**建议修复方向：**

- 优先保证本地状态落库成功
- 远程同步改为可靠消息或补偿机制
- 更理想方案是引入 outbox、本地消息表或事务消息

### 3. 库存接口缺少边界校验

**问题等级：高**

**涉及文件：**

- [CourseController.java](D:\vis\yunh\yunh-service\yunh-service-course\src\main\java\com\yunh\course\controller\CourseController.java:147)
- [CourseController.java](D:\vis\yunh\yunh-service\yunh-service-course\src\main\java\com\yunh\course\controller\CourseController.java:156)
- [CourseStockServiceImpl.java](D:\vis\yunh\yunh-service\yunh-service-course\src\main\java\com\yunh\course\service\impl\CourseStockServiceImpl.java:35)
- [CourseMapper.java](D:\vis\yunh\yunh-service\yunh-service-course\src\main\java\com\yunh\course\mapper\CourseMapper.java:18)

**问题描述：**

库存相关接口缺少对 `count`、`courseId` 等参数的有效性检查。

存在以下风险场景：

- `count = -1` 时，可能反向增加库存
- `count = 0` 时，产生无意义操作
- `count = null` 时，存在空指针风险

**潜在后果：**

- 库存逻辑被绕过
- 出现错误加库存
- 接口稳定性下降

**建议修复方向：**

- 控制器层做非空与正数校验
- 服务层再做一次保护性校验
- 使用 DTO 配合 `@Min(1)`、`@NotNull`

### 4. 登出逻辑可能假成功

**问题等级：高**

**涉及文件：**

- [AuthService.java](D:\vis\yunh\yunh-auth\src\main\java\com\yunh\auth\service\AuthService.java:171)
- [AuthService.java](D:\vis\yunh\yunh-auth\src\main\java\com\yunh\auth\service\AuthService.java:193)
- [AuthController.java](D:\vis\yunh\yunh-auth\src\main\java\com\yunh\auth\controller\AuthController.java:71)

**问题描述：**

登出过程中，如果 Redis 写黑名单失败、refresh token 删除失败、JWT 解析失败等异常被吞掉，接口仍可能返回成功。

**潜在后果：**

- 用户认为自己已退出
- 实际 token 仍然有效
- 系统存在认证绕过和会话残留风险

**建议修复方向：**

- 不要统一吞掉所有异常
- 黑名单写入失败时应显式返回失败
- 日志中应记录完整上下文

- [AuthController.java](D:\vis\yunh\yunh-auth\src\main\java\com\yunh\auth\controller\AuthController.java:54)
- [UserController.java](D:\vis\yunh\yunh-service\yunh-service-user\src\main\java\com\yunh\user\controller\UserController.java:75)

- [SelectCourseDTO.java](D:\vis\yunh\yunh-pojo\yunh-pojo-common\src\main\java\com\yunh\pojo\dto\SelectCourseDTO.java:18)
- [VideoController.java](D:\vis\yunh\yunh-service\yunh-service-video\src\main\java\com\yunh\video\controller\VideoController.java:46)

**问题描述：**

多个接口对空请求体、空字段、非法字段值缺乏统一校验，参数可以直接流入业务层。

**潜在后果：**

- 容易触发 NPE

- 无效数据进入数据库

**建议修复方向：**

- 全面使用 DTO
- 引入 `@Validated`、`@Valid`
- 统一处理参数校验异常

### 6. 状态字段比较存在空指针风险

**问题等级：中**

**涉及文件：**

- [UserCourseService.java](D:\vis\yunh\yunh-service\yunh-service-user\src\main\java\com\yunh\user\service\UserCourseService.java:154)
- [UserCourseService.java](D:\vis\yunh\yunh-service\yunh-service-user\src\main\java\com\yunh\user\service\UserCourseService.java:159)
- [UserCourseService.java](D:\vis\yunh\yunh-service\yunh-service-user\src\main\java\com\yunh\user\service\UserCourseService.java:200)
- [OrderServiceImpl.java](D:\vis\yunh\yunh-service\yunh-service-order\src\main\java\com\yunh\order\service\impl\OrderServiceImpl.java:66)
- [OrderServiceImpl.java](D:\vis\yunh\yunh-service\yunh-service-order\src\main\java\com\yunh\order\service\impl\OrderServiceImpl.java:82)

**问题描述：**

多个业务流程中直接对 `Integer status` 进行比较，若状态字段为空会发生隐式拆箱并抛出空指针异常。

**潜在后果：**

- 支付、取消、超时关闭等流程直接报错
- 历史脏数据或异常数据放大风险

**建议修复方向：**

- 使用 `Objects.equals(status, 1)` 这类方式比较
- 状态字段在表结构和实体层统一设置默认值和非空约束

### 7. RocketMQ 事务消息实现不可靠

**问题等级：中**

**涉及文件：**

- [CourseOrderTransactionListener.java](D:\vis\yunh\yunh-service\yunh-service-user\src\main\java\com\yunh\user\mq\transaction\CourseOrderTransactionListener.java:33)
- [CourseOrderTransactionListener.java](D:\vis\yunh\yunh-service\yunh-service-user\src\main\java\com\yunh\user\mq\transaction\CourseOrderTransactionListener.java:44)
- [CourseOrderTransactionListener.java](D:\vis\yunh\yunh-service\yunh-service-user\src\main\java\com\yunh\user\mq\transaction\CourseOrderTransactionListener.java:73)

**问题描述：**

事务消息监听器当前仅用内存结构记录事务状态，且本地事务逻辑并未真正完整落地。

这类实现一旦遇到：

- 服务重启
- Broker 回查
- 多实例部署

就容易出现状态丢失和误判。

**潜在后果：**

- 回查结果不稳定
- 消息误提交或误回滚
- 后续上线后产生隐蔽性故障

**建议修复方向：**

- 事务状态持久化到数据库
- 补全真实的本地事务逻辑
- 如果只是演示代码，应明确隔离标记

## 三、工程层面观察

### 1. 自动化测试基本缺失

**问题等级：高**

**问题描述：**

当前项目基本未发现有效业务测试目录，现有检索结果中的测试文件主要来自前端 `node_modules`，说明项目自身的单元测试、集成测试、接口测试较弱甚至缺失。

**潜在后果：**

- 修改风险高
- 回归成本高
- 分布式事务、消息队列、鉴权链路难以稳定验证

**建议修复方向：**

- 优先为核心链路补最小测试集
- 从认证、选课、支付、库存四个链路先补

---

### 2. 当前目录不是 Git 仓库根目录

**问题等级：中**

**问题描述：**

在项目根执行 `git status --short` 时返回：

```text
fatal: not a git repository (or any of the parent directories): .git
```

这说明当前工作目录并不是 Git 仓库根，或者 `.git` 不在该目录。

**潜在后果：**

- 版本管理信息缺失
- 无法直接查看变更、分支、提交记录
- 后续协作和审计不方便

**建议修复方向：**

- 确认真实 Git 根目录
- 确认当前目录是否为导出副本

---

### 3. 仓库中混入较多运行产物和本地环境文件

**问题等级：中**

**问题描述：**

当前项目根目录存在较多运行期或本地环境目录，例如：

- `log/`
- `.idea/`
- `.vscode/`
- `yunh-admin/node_modules`
- `config/` 下的中间件运行资源

**潜在后果：**

- 仓库体积膨胀
- 提交内容不干净
- 环境可移植性下降

**建议修复方向：**

- 清理不应纳入版本管理的目录
- 完善 `.gitignore`
- 区分源码、文档、依赖、运行产物

## 四、优先治理建议

建议按以下顺序处理：

### 第一批：立即处理

- 选课链路远程调用结果强校验
- 支付链路状态一致性修复
- 库存接口参数边界校验
- 登出逻辑异常处理修复

### 第二批：体系化治理

- 全局统一参数校验
- 全局统一异常处理
- 状态字段非空化和比较方式修正
- MQ 事务消息实现重构

### 第三批：工程补强

- 建立最小自动化测试集
- 清理仓库运行产物
- 明确 Git 根目录和协作规范

---

## 五、结论

当前项目已经具备完整的微服务学习与演练价值，功能覆盖面较广，架构层次也比较清晰。

但从工程质量和上线稳定性角度看，当前最大的短板不是“功能没做”，而是：

- **失败路径没有严格收口**
- **跨服务一致性没有形成可靠保障**
- **参数与异常处理体系还不统一**
- **测试体系明显不足**

如果后续准备继续推进，建议优先围绕“选课链路”和“支付链路”做一轮集中修复，因为这两条链路当前承载了项目中最明显的高风险一致性问题。
