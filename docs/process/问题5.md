# 项目新增问题清单（第五版）

## 1. 文档说明

本文档只记录 **相对于 `docs/process/问题4.md` 新发现的问题**，不重复以下内容：

- `问题4.md` 已经明确记录的通用问题
- 前几轮已经修复完成的问题
- 已经纳入当前主线整改的问题

本轮检查重点放在：

- 小程序支付链路
- 支付服务与退款服务
- 管理端支付页面
- 仓库中的前端构建产物

---

## 2. 新增高优问题

### 2.1 小程序支付创建接口与后端参数绑定方式不一致

**类别：前后端接口一致性**  
**优先级：P1**

**问题说明：**

- 小程序 `services/pay.js` 使用 `POST + application/json` 方式调用 `/api/pay/create`
- 后端 `PayController.createPayment(...)` 仍使用 `@RequestParam` 接收 `orderNo`、`amount`、`payType`
- 当前请求封装不会自动把 JSON body 转成 query string

**涉及文件：**

- `yunh-mini/miniprogram/services/pay.js`
- `yunh-mini/miniprogram/utils/request.js`
- `yunh-service/yunh-service-pay/src/main/java/com/yunh/pay/controller/PayController.java`

**风险：**

- 支付单创建请求可能直接因参数绑定失败而报错
- 订单创建成功但支付单未创建，导致支付页流程中断

---

### 2.2 支付回调被前端直接暴露，后端也未真正限制内部来源

**类别：安全 / 业务一致性**  
**优先级：P0**

**问题说明：**

- 小程序结果页直接调用 `payService.payCallback(this.data.paymentNo)`
- 小程序 `services/pay.js` 会伪造 `thirdPartyNo = WX-DEMO-${Date.now()}`
- 后端 `PayController.payCallback(...)` 虽然声明了 `X-Internal-Call`，但没有做任何校验
- 网关目前也没有把 `/api/pay/callback` 放进内部专用路径限制

**涉及文件：**

- `yunh-mini/miniprogram/packageOrder/pages/result.js`
- `yunh-mini/miniprogram/services/pay.js`
- `yunh-service/yunh-service-pay/src/main/java/com/yunh/pay/controller/PayController.java`
- `yunh-support/yunh-service-gateway/src/main/java/com/yunh/gateway/filter/AuthGlobalFilter.java`

**风险：**

- 已登录用户可直接伪造支付成功
- 支付服务与订单服务会把未真实付款的订单标记为已支付
- 整条支付链路仍停留在“演示可用”而非“真实可控”的状态

---

### 2.3 支付金额仍完全信任前端传值，订单服务也未校验实付金额

**类别：安全 / 业务一致性**  
**优先级：P0**

**问题说明：**

- 小程序确认订单页直接把页面里的课程价格作为 `amount` 传给 `/api/pay/create`
- `PayController.createPayment(...)` 只校验“有没有用户头”，不校验该订单是否属于当前用户，也不校验金额是否与订单应付金额一致
- `PaymentServiceImpl.createPayment(...)` 直接落库前端传入的 `amount`
- `OrderServiceImpl.payOrder(...)` 只根据 `orderNo` 把订单状态改成已支付，没有任何实付金额比对

**涉及文件：**

- `yunh-mini/miniprogram/packageOrder/pages/confirm.js`
- `yunh-service/yunh-service-pay/src/main/java/com/yunh/pay/controller/PayController.java`
- `yunh-service/yunh-service-pay/src/main/java/com/yunh/pay/service/impl/PaymentServiceImpl.java`
- `yunh-service/yunh-service-order/src/main/java/com/yunh/order/service/impl/OrderServiceImpl.java`

**风险：**

- 前端篡改金额后仍可能完成“支付成功”链路
- 订单金额、支付金额、对账结果出现错位
- 后续退款、统计、财务核对都会受到影响

---

### 2.4 退款申请与退款查询仍缺少角色和归属校验

**类别：权限**  
**优先级：P0**

**问题说明：**

- `PayController.refund(...)` 和 `PayController.getRefund(...)` 只接收了 `X-Role`，但并未实际执行管理员限制
- 接口也没有基于 `X-User-Id` 校验订单是否属于当前用户
- 当前只要知道 `orderNo`，就具备进一步探测和操作退款记录的可能

**涉及文件：**

- `yunh-service/yunh-service-pay/src/main/java/com/yunh/pay/controller/PayController.java`
- `yunh-service/yunh-service-pay/src/main/java/com/yunh/pay/service/impl/RefundServiceImpl.java`
- `yunh-admin/src/api/pay.js`

**风险：**

- 越权查询他人退款记录
- 越权发起退款申请
- 后台页面与普通用户接口边界混淆

---

## 3. 新增中优问题

### 3.1 退款记录写入字段不完整，管理端字段读取也不一致

**类别：前后端接口一致性 / 数据完整性**  
**优先级：P1**

**问题说明：**

- `Refund` 实体包含 `refundAmount`、`userId`、`paymentNo`、`status` 等字段
- `RefundServiceImpl.applyRefund(...)` 实际只写入了 `orderNo`、`reason`、`status`、时间字段
- 管理端 `PaymentList.vue` 展示时读取的是 `refundAmount` 和 `refundStatus`
- 后端实体字段名实际是 `status`，而不是 `refundStatus`

**涉及文件：**

- `yunh-pojo/yunh-pojo-pay/src/main/java/com/yunh/pay/pojo/Refund.java`
- `yunh-service/yunh-service-pay/src/main/java/com/yunh/pay/service/impl/RefundServiceImpl.java`
- `yunh-admin/src/views/PaymentList.vue`

**风险：**

- 退款金额可能长期为空
- 退款状态在管理端显示错误或直接显示异常
- 后续审计、对账、售后追踪信息不完整

---

### 3.2 支付记录的 `userId` 解析失败后可能生成脏数据，并触发后续空指针风险

**类别：稳定性 / 数据完整性**  
**优先级：P1**

**问题说明：**

- `PaymentServiceImpl.resolveUserId(orderNo)` 在远程调用失败时直接返回 `null`
- `createPayment(...)` 没有对这个结果做失败收口，仍可能继续保存支付记录
- `PayController.getPayStatus(...)` 后续又直接执行 `payment.getUserId().equals(...)`

**涉及文件：**

- `yunh-service/yunh-service-pay/src/main/java/com/yunh/pay/service/impl/PaymentServiceImpl.java`
- `yunh-service/yunh-service-pay/src/main/java/com/yunh/pay/controller/PayController.java`

**风险：**

- 生成 `userId` 为空的支付脏数据
- 查询支付状态时出现空指针异常
- 权限判断逻辑被脏数据绕乱

---

## 4. 新增工程问题

### 4.1 管理端 `dist` 构建产物直接进入仓库，且存在与源码脱节风险

**类别：工程治理**  
**优先级：P2**

**问题说明：**

- 仓库内存在完整的 `yunh-admin/dist` 构建产物
- 这些文件是已编译后的静态资源，不是源码
- 当源码已经修复而 `dist` 没同步重建时，仓库里会同时存在两套行为定义

**涉及路径：**

- `yunh-admin/dist/index.html`
- `yunh-admin/dist/assets/*`

**风险：**

- 代码审查时容易误判实际运行逻辑
- 若部署流程误用仓库内旧 `dist`，会把已修复问题重新带回环境
- 增加协作、比对和发布成本

---

## 5. 建议处理顺序

建议新增问题按以下顺序推进：

1. **立即封住 `/api/pay/callback` 的前端直连能力**  
   改为内部专用回调、服务间签名校验或明确的内网路由。

2. **统一支付创建接口契约**  
   前后端统一参数传递方式，同时在服务端强制校验订单归属和应付金额。

3. **补齐退款接口的权限与数据完整性**  
   退款申请、退款查询、退款展示字段要一起修，不要只修其中一层。

4. **阻断支付脏数据落库**  
   `resolveUserId` 失败时应直接失败返回，不应继续创建支付记录。

5. **清理管理端构建产物入库问题**  
   明确 `dist` 是否应该入库，并配套 `.gitignore` 或发布规范。

---

## 6. 结论

相对 `问题4.md`，本轮最重要的新增发现集中在 **支付与退款链路**：

- **支付创建接口当前就存在契约不一致**
- **支付回调仍可被前端直接伪造**
- **退款权限与退款数据模型同时存在缺口**

这说明项目当前剩余风险已经不只是“通用安全收口”，而是开始进入 **关键业务链路的精确修补阶段**。  
其中优先级最高的是支付回调、支付金额校验和退款权限三项。
