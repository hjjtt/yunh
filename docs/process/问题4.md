# 项目问题总台账（第四版）

## 1. 文档说明

本文档用于汇总当前阶段的项目问题，并统一整理以下几类信息：

- 之前已经发现、**现在已解决**的问题
- 当前仍然存在、**需要继续处理**的问题
- 项目级的长期短板与优化方向

本版是在以下文档基础上合并整理的：

- `docs/process/问题.md`
- `docs/process/问题2.md`
- `docs/process/问题3.md`

同时结合了本轮新增的前端审查、项目问题审查和项目总结结果。

---

## 2. 当前总体判断

项目整体已经不是“搭骨架”的阶段，而是进入了“**从可运行走向可交付**”的阶段。

当前项目具备以下核心能力：

- Spring Cloud 多模块微服务架构
- 微信小程序端
- Vue 管理端
- Nacos 注册中心与配置中心
- Gateway 网关
- JWT 鉴权
- OpenFeign 服务调用
- Sentinel 限流
- Seata 分布式事务
- RocketMQ 消息队列

当前最大的矛盾已经不是“功能有没有”，而是：

- **安全边界是否真正收口**
- **配置和文档是否只有一个可信标准**
- **前后端风格和成熟度是否一致**
- **修复后是否具备稳定回归能力**

---

## 3. 已解决问题

以下问题已在前几轮处理中完成修复或明显缓解，不再作为当前最高优先级问题处理。

### 3.1 页面显示与交互问题已修复

**已解决内容：**

- `packageProfile/pages/comment-create` 页面发布按钮不可见
- `packageProfile/pages/question-create` 页面发布按钮不可见
- `packageProfile/pages/my-comments`、`packageProfile/pages/my-questions` 按钮不清晰
- `pages/auth/login` 登录页布局显示异常
- 提问标题输入区显示异常

**当前状态：**

- 已完成页面样式重构
- 相关页面的按钮层级、输入框显示和整体布局已改善

---

### 3.2 用户敏感信息公开暴露已开始收口

**已解决/已缓解内容：**

- 用户公开接口不再继续直接以完整用户实体作为前端安全返回模型
- 小程序登录态本地存储已增加安全字段过滤

**涉及修复方向：**

- 用户公开出参改为安全视图对象
- 本地 `session` 过滤 `password/email/phone` 等不应缓存的字段

**当前状态：**

- **已明显缓解**
- 仍建议后续做一次完整回归验证，确认所有旧缓存和所有调用链都已切换完成

---

### 3.3 网关课程白名单过宽已开始收口

**已解决/已缓解内容：**

- 原先 `/api/course/**` 全量放行的风险已经开始处理
- 教师端课程相关请求已补充显式鉴权标记

**当前状态：**

- **已明显缓解**
- 仍建议后续结合真实联调再验证一次课程创建、更新、删除、库存相关接口的访问边界

---

## 4. 当前仍需处理的高优问题

以下问题是当前真正还没有完全解决、并且优先级最高的部分。

### 4.1 验证码机制仍然失效

**类别：安全**  
**优先级：P0**

**问题说明：**

- 后端直接把 `captchaCode` 返回给客户端
- 前端也直接显示验证码答案

**涉及文件：**

- `yunh-auth/src/main/java/com/yunh/auth/service/AuthService.java`
- `yunh-auth/src/main/java/com/yunh/auth/controller/AuthController.java`
- `yunh-mini/miniprogram/pages/auth/login.js`

**风险：**

- 验证码形同虚设
- 脚本可直接绕过人机校验

---

### 4.2 登出后 token 仍可继续使用

**类别：安全**  
**优先级：P0**

**问题说明：**

- 认证服务已经有 token 黑名单逻辑
- 但网关鉴权时还没有消费黑名单状态

**涉及文件：**

- `yunh-auth/src/main/java/com/yunh/auth/service/AuthService.java`
- `yunh-support/yunh-service-gateway/src/main/java/com/yunh/gateway/filter/AuthGlobalFilter.java`

**风险：**

- 用户退出登录后，旧 token 仍可能继续可用直到自然过期

---

### 4.3 订单 / 选课 / 支付仍存在越权入口

**类别：权限**  
**优先级：P0**

**问题说明：**

- 接口仍信任路径参数或请求参数中的 `userId`、`orderId`、`orderNo`
- 没有完全以当前登录用户身份作为唯一可信来源

**涉及文件：**

- `yunh-service/yunh-service-order/src/main/java/com/yunh/order/controller/OrderController.java`
- `yunh-service/yunh-service-user/src/main/java/com/yunh/user/controller/UserController.java`

**风险：**

- 查询他人订单
- 替他人选课
- 替他人支付

---

### 4.4 互动模块资源归属仍未完全收口

**类别：权限**  
**优先级：P1**

**问题说明：**

- 创建时已经开始使用 `X-User-Id`
- 但删除、采纳、查看个人笔记等接口仍缺少资源归属校验

**涉及文件：**

- `yunh-service/yunh-service-interaction/src/main/java/com/yunh/interaction/controller/CommentController.java`
- `yunh-service/yunh-service-interaction/src/main/java/com/yunh/interaction/controller/QuestionController.java`
- `yunh-service/yunh-service-interaction/src/main/java/com/yunh/interaction/controller/AnswerController.java`
- `yunh-service/yunh-service-interaction/src/main/java/com/yunh/interaction/controller/NoteController.java`

**风险：**

- 删除他人内容
- 采纳不属于自己的回答
- 读取不属于自己的笔记

---

### 4.5 配置基线仍然不可信

**类别：配置**  
**优先级：P1**

**问题说明：**

- 初始化 SQL 是多库模型
- Nacos 本地快照是多库模型
- 配置发布脚本却可能按单库模型发布
- 同时仓库里还保留了较多默认账号密码和历史环境痕迹

**涉及文件：**

- `database/init.sql`
- `config/publish-nacos-configs.ps1`
- `config/nacos-server-3.1.1/nacos/data/tenant-config-data/public/DEFAULT_GROUP/*.yml`

**风险：**

- 同一项目存在多套“真实标准”
- 本地环境、脚本、文档、运行结果可能互相打架

---

## 5. 仍需关注的业务一致性问题

这些问题未必比安全问题更急，但它们直接影响项目稳定性和后续可靠交付。

### 5.1 选课链路远程扣库存结果校验不足

**类别：业务一致性**  
**优先级：P1**

**问题说明：**

- 用户选课主流程中，远程扣库存失败时没有形成足够严格的失败收口

**风险：**

- 本地选课记录已创建
- 课程库存未同步扣减
- Seata / MQ 链路出现状态不一致

---

### 5.2 支付回调存在跨服务状态错位风险

**类别：业务一致性**  
**优先级：P1**

**问题说明：**

- 支付回调处理里，本地支付状态和远程订单状态更新顺序不够稳妥

**风险：**

- 订单服务与支付服务状态不一致
- 幂等、退款、对账成本变高

---

### 5.3 RocketMQ 事务消息实现仍偏演示化

**类别：稳定性**  
**优先级：P2**

**问题说明：**

- 事务消息状态更多依赖内存结构
- 更像教学示例，离生产级可靠实现还有差距

**风险：**

- 服务重启后状态丢失
- Broker 回查结果不稳定

---

### 5.4 课程详情接口输出完整 Authorization 日志

**类别：安全 / 日志治理**  
**优先级：P2**

**问题说明：**

- 课程详情接口日志里仍可能输出完整 Bearer Token

**风险：**

- 日志泄露后可直接复用登录态

---

## 6. 前端与产品层问题

这部分不一定最危险，但会明显影响项目的完成度观感和产品一致性。

### 6.1 小程序整体风格已经形成，但成熟度不均

**当前判断：**

- `yunh-mini` 已经形成比较鲜明的“复古学院派”视觉语言
- 主题、纹理、标题、卡片系统已经有辨识度

**做得较好的页面：**

- `yunh-mini/miniprogram/pages/home/index`
- `yunh-mini/miniprogram/pages/auth/login`
- `yunh-mini/miniprogram/pagesTeacher/home/index`

**仍需加强的页面：**

- `yunh-mini/miniprogram/packageCourse/pages/detail`
- `yunh-mini/miniprogram/packageCourse/pages/player`
- 部分学员端页面仍有硬编码颜色和细节不统一问题

---

### 6.2 管理端视觉系统明显落后于小程序端

**类别：前端美观 / 品牌一致性**  
**优先级：P2**

**问题说明：**

- 小程序端已有明确主题风格
- `yunh-admin` 仍然偏默认后台模板风
- 两端看起来像两套产品

**最需要优化的页面：**

- `yunh-admin/src/views/Login.vue`
- `yunh-admin/src/views/CourseList.vue`
- `yunh-admin/src/layout/Layout.vue`

**建议方向：**

- 先统一管理端全局视觉变量和布局层
- 再统一登录页和主要列表页风格

---

### 6.3 前后端 API 一致性仍有补齐空间

**类别：前端 / 接口治理**  
**优先级：P2**

**问题说明：**

- 搜索、购物车等能力还存在前后端对接不充分的问题
- 部分服务层函数定义重复或职责边界不够清晰

**风险：**

- 功能明明“有后端”但前端仍未真正接通
- 后续维护成本上升

---

## 7. 工程与交付问题

### 7.1 文档与实现仍不一致

**类别：文档**  
**优先级：P1**

**问题说明：**

- 根 README 的端口、阶段说明、文件引用已经明显过时
- 小程序 README 和实际配置不完全一致

**风险：**

- 新人上手成本高
- 环境搭建和联调判断容易偏差

---

### 7.2 自动化测试仍然不足

**类别：测试**  
**优先级：P1**

**问题说明：**

- 目前缺少最小的稳定 smoke test
- 大多数验证仍依赖人工联调和日志回看

**风险：**

- 修复后难确认是否引入回归
- 分布式链路稳定性缺乏客观保障

---

### 7.3 仓库洁净度仍然偏差

**类别：工程治理**  
**优先级：P2**

**问题说明：**

- `target/`
- `node_modules/`
- `log/`
- `.idea/`
- `.vscode/`
- 中间件运行目录

等内容仍混在工作目录中。

**风险：**

- 干扰排查和审查
- 增加误判和协作成本

---

## 8. 当前建议优先级

建议下一步按以下顺序继续推进：

### 第一批：安全收口

- 验证码返回模型改造
- token 黑名单接入网关鉴权
- 订单 / 选课 / 支付接口统一基于当前登录用户校验
- 互动模块资源归属校验

### 第二批：运行标准统一

- 明确唯一数据库模型
- 明确 Nacos 配置来源
- 重写 README 和启动手册
- 清理默认账号密码和历史环境残留

### 第三批：工程化补强

- 建立最小 smoke test
- 清理运行产物和无效目录
- 统一管理端视觉系统

---

## 9. 结论

如果把当前项目放在一个更现实的维度上看，它已经完成了“从 0 到 1”的核心搭建：

- 架构成型
- 主链路成型
- 前后端成型
- 分布式基础设施已接入

但它还没有完成“从可运行到可交付”的最后一段收口。

当前最核心的任务，不是继续堆新功能，而是把下面四件事做扎实：

- **安全边界收口**
- **配置标准统一**
- **验证闭环建立**
- **产品和品牌一致性提升**

这也是 `问题4.md` 作为总台账的主要结论。
