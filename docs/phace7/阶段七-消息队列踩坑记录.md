# 阶段七 - 消息队列（RocketMQ）踩坑记录

## 环境信息

| 项目 | 版本 |
|------|------|
| RocketMQ | 4.9.4 |
| rocketmq-spring-boot-starter | 2.2.3 |
| rocketmq-client | 5.0.0（starter 传递依赖） |
| JDK | 20 / 21 |
| 操作系统 | Windows 11 |

## 踩坑一：ROCKETMQ_HOME 环境变量未设置

### 错误信息
```
Please set the ROCKETMQ_HOME variable in your environment!
```

### 原因
RocketMQ 的 `mqadmin.cmd`、`mqbroker.cmd`、`mqnamesrv.cmd` 都依赖 `ROCKETMQ_HOME` 环境变量来定位安装目录。Windows 系统默认没有设置这个变量。

### 解决
在 PowerShell 中临时设置：
```powershell
$env:ROCKETMQ_HOME = "d:\vis\yunh\config\rocketmq-all-4.9.4-bin-release"
```
如需永久生效，可在系统环境变量中添加。

## 踩坑二：CLASSPATH 包含空格路径导致启动失败

### 错误信息
```
错误: 找不到或无法加载主类 Files\Java\jdk-21.0.10\lib\dt.jar;C:\Program
```

### 原因
Windows 系统 CLASSPATH 环境变量包含 `C:\Program Files\Java\...` 路径。RocketMQ 的 `runserver.cmd` 和 `runbroker.cmd` 原始脚本通过以下方式拼接：
```batch
set "CLASSPATH=.;%BASE_DIR%conf;%BASE_DIR%lib\*;%CLASSPATH%"
```
这会把系统 CLASSPATH 追加进来，而 `-cp %CLASSPATH%` 没有加引号，空格导致路径被截断。

### 解决
修改 `runserver.cmd` 和 `runbroker.cmd`：
1. **去掉系统 CLASSPATH 继承**：
```batch
set "CLASSPATH=.;%BASE_DIR%conf;%BASE_DIR%lib\*"
```

2. **把 `-cp` 从 JAVA_OPT 移到 java 命令行，并加引号**：
```batch
"%JAVA%" %JAVA_OPT% -cp "%CLASSPATH%" %*
```

## 踩坑三：JDK 20/21 不兼容 CMS GC 和已废弃参数

### 错误信息
```
Unrecognized VM option 'UseConcMarkSweepGC'
Unrecognized VM option 'UseBiasedLocking'
```

### 原因
RocketMQ 4.9.4 默认脚本使用 CMS 垃圾回收器和 `UseBiasedLocking` 参数，这些在 JDK 15+ 已被移除。

### 解决
- CMS → G1GC：`-XX:+UseG1GC`
- 移除 `UseBiasedLocking`
- GC 日志格式从 `-XX:+PrintGCDetails` 改为 `-Xlog:gc*`
- 内存从 4g/8g 缩减到 1g（开发环境够用）

## 踩坑四：Broker 启动未加载 broker.conf（核心问题）

### 错误信息
```
org.apache.rocketmq.client.exception.MQClientException: No route info of this topic: course_order_topic
```

### 原因
这是**最关键**的一个坑。启动 Broker 时使用的命令是：
```powershell
mqbroker.cmd
```
这个命令**没有指定 `-c broker.conf` 参数**，导致 Broker 启动时完全没有加载我们的配置文件。

后果：
- `namesrvAddr = 127.0.0.1:9876` 未生效 → Broker 可能用了默认值或未正确连接 NameServer
- `autoCreateTopicEnable = true` 未生效（4.9.x 默认为 false）→ Topic 不会自动创建
- 即使手动用 `mqadmin updateTopic` 创建了 Topic，由于 Broker 和 NameServer 的连接配置不一致，路由信息也无法正确传递

### 排查过程
通过查看 Broker 进程的完整命令行确认：
```powershell
(Get-CimInstance Win32_Process -Filter "ProcessId = 30292").CommandLine
```
输出末尾只有 `org.apache.rocketmq.broker.BrokerStartup`，没有 `-c` 参数。

### 解决
**必须**用以下命令启动 Broker，加上 `-c` 指定配置文件：
```powershell
$env:ROCKETMQ_HOME = "d:\vis\yunh\config\rocketmq-all-4.9.4-bin-release"
mqbroker.cmd -c d:\vis\yunh\config\rocketmq-all-4.9.4-bin-release\conf\broker.conf
```

启动成功后日志应显示：
```
The broker[broker-a, 10.150.4.187:10911] boot success. serializeType=JSON and name server is 127.0.0.1:9876
```

> ⚠️ 如果看不到 `name server is 127.0.0.1:9876`，说明配置文件没有被加载。

## 踩坑五：rocketmq-spring 2.2.3 API 与文档不一致

### 错误一：`@RocketMQTransactionListener` 没有 `txProducerGroup` 属性

```java
// 报错：找不到符号 txProducerGroup()
@RocketMQTransactionListener(txProducerGroup = "xxx")
```

**原因**：rocketmq-spring-boot-starter 2.2.3 搭配的 rocketmq-client 5.0.0 中，`@RocketMQTransactionListener` 注解只有 `rocketMQTemplateBeanName` 属性，没有 `txProducerGroup`。

**解决**：去掉属性，使用默认值：
```java
@RocketMQTransactionListener
```

### 错误二：`Message<?>` 与 `Message` 类型冲突

```java
// 报错：same erasure, neither overrides the other
@Override
public RocketMQLocalTransactionState executeLocalTransaction(Message<?> msg, Object arg) {
```

**原因**：`RocketMQLocalTransactionListener` 接口使用的是原始类型 `Message`（raw type），不是 `Message<?>`。JDK 21 编译时对泛型检查更严格。

**解决**：使用原始类型并加 `@SuppressWarnings`：
```java
@SuppressWarnings({"rawtypes", "unchecked"})
@Override
public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
```

## 踩坑六：重复选课导致唯一索引冲突

### 错误信息
```
DuplicateKeyException: Duplicate entry '1-2' for key 't_user_course.uk_user_course'
```

### 原因
测试时多次用相同参数（userId=1, courseId=2）调用选课接口，数据库有唯一索引限制。

### 解决
每次测试前清理数据：
```sql
DELETE FROM yunh_user.t_user_course WHERE user_id=1 AND course_id=2;
```

## 最终正确的启动顺序

```
1. 启动 Nacos        → nacos/bin/startup.cmd -m standalone
2. 启动 Seata Server  → seata/bin/seata-server.bat
3. 启动 RocketMQ NameServer → mqnamesrv.cmd
4. 启动 RocketMQ Broker     → mqbroker.cmd -c conf/broker.conf  ← 必须加 -c
5. 启动 yunh-service-user
6. 启动 yunh-service-course
```
